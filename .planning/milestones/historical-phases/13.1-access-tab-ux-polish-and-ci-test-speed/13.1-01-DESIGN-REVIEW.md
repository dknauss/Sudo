# Phase 13.1 Plan 01 Design Brief

Operators currently have to type raw numeric user IDs and decode governance capability slugs on the Settings → Sudo → Access grant form, which creates an avoidable operator-error risk in a security-sensitive capability-grant path; this polish is presentation-only and must not change authorization. The proposed approach is to keep the existing AJAX contract and server handler intact while replacing the user field in `includes/class-admin.php::render_access_tab()` with a server-rendered `<select id="wp-sudo-grant-user">` populated by `get_users( array( 'role' => 'administrator', 'orderby' => 'display_name', 'order' => 'ASC', 'fields' => 'all', 'number' => -1 ) )`, starting with `<option value="0">— Select a user —</option>`. A `CAP_LABELS` constant next to `GOVERNANCE_CAPS` will render plain-English option text with the slug in parentheses and the raw slug in `title`, while each capability `<option value>` remains the exact canonical slug. The only JavaScript-facing text change is the localized `invalidUser` string, because the existing `parseInt( userInput.value, 10 )` guard, `focus()`, `wp.a11y.speak`, nonce, action name, and request shape already work for a `<select>`. The change must not modify `handle_grant_cap()`, `handle_revoke_cap()`, `(int) $_POST['user_id']`, `check_ajax_referer( 'wp_sudo_access', '_nonce' )`, AJAX action names, or `GOVERNANCE_CAPS` validation. `render_access_tab()` is multisite-aware here because WordPress `get_users()` is blog-scoped by role, so the picker intentionally lists current-blog administrators. Open question, recorded for later rather than implemented now: whether the picker should support broader eligibility than administrator-role users.

## Reviewer critique

The design reviewer flagged the admin-only picker as a UI reachability change rather than a server authorization change: direct AJAX remains capable of granting to any valid user ID, while the primary form lists current-blog administrators only. The implementation therefore keeps this framed as a convenience picker, not a security boundary or policy enforcement layer, and records broader eligibility for custom roles/direct-cap users as deferred. The reviewer also called out multisite scope clarity, drift-panel eligibility mismatch, large-site scalability, empty-state copy, user-label ambiguity, CAP_LABELS drift risk, and accessibility copy; the implementation keeps the locked admin-role scope, includes display name plus login for disambiguation, updates visible/help/error text to selection language, uses slug-keyed labels with slug fallback, and preserves direct AJAX/drift-panel semantics. No handler, nonce, action-name, request-shape, or `GOVERNANCE_CAPS` validation changes are allowed or needed.
