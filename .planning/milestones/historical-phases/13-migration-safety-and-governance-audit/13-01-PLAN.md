<objective>
Land the only new production code in Phase 13: the `upgrade_4_0_0()` migration routine
that deletes the now-inert `wp_sudo_governance_mode` option on the 3.x -> 4.0.0 boundary,
a `cleanup_inert_governance_mode_option()` admin_init hook that self-heals the option on
the next admin load even when the version is already stamped 4.0.0, and a reworked
"fixed" confirmation notice with a semantically correct developer signal. Also lock the
capability audit (MIG-05) at the unit level: assert admin surfaces route through
`wp_sudo_can('manage_wp_sudo')`, not bare `manage_options`.

Purpose: Make the Phase 12 persistent migration notice truthful. Because the option is
now auto-deleted (on upgrade AND on detection), the notice becomes a one-time, brief,
dismissible "we fixed a leftover setting" confirmation rather than a persistent nag with
misleading manual-removal instructions.

Output: `upgrade_4_0_0()` routine, `cleanup_inert_governance_mode_option()` method,
reworked `render_compatibility_mode_notice()`, audit-hook dev signal, and updated unit
tests encoding the new contract.
</objective>
