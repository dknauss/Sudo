**KEYB-03 — Ctrl+Shift+S navigates to challenge when no session:**

Source notes to include: admin/js/wp-sudo-shortcut.js — keydown handler checks e.shiftKey && (e.ctrlKey || e.metaKey) && e.key.toLowerCase() === 's', then sets window.location.href = config.challengeUrl (verified). Only enqueued when session is NOT active and not on the challenge page itself. Config object: window.wpSudoShortcut (localized via wp_localize_script). challengeUrl includes return_url= the current admin page.

- Ensure no active sudo session before this test (no beforeEach needed since activateSudoSession is only called in KEYB-04; KEYB-03 navigates without one).
- Navigate to '/wp-admin/' (dashboard — NOT the challenge page, which excludes the shortcut script).
- Wait for wpSudoShortcut config: `page.waitForFunction(() => typeof (window as any).wpSudoShortcut !== 'undefined' && !!(window as any).wpSudoShortcut.challengeUrl)`
- Use Promise.all: waitForURL(/page=wp-sudo-challenge/, timeout 10_000) + keyboard.press('Control+Shift+S').
- Assert page.locator('#wp-sudo-challenge-card').toBeVisible().

Pitfall to document: Shortcut script is NOT loaded on the challenge page itself (class-plugin.php guard: if 'wp-sudo-challenge' === $page return early). Always start KEYB-03 from the dashboard. Use Control+Shift+S (not Meta+Shift+S) — tests run on Linux CI where ctrlKey is the modifier. The JS checks e.ctrlKey || e.metaKey so Control works on Linux and Meta works on macOS; test uses Control for CI consistency.

**KEYB-04 — Ctrl+Shift+S flashes admin bar when session is active:**

Source notes to include: admin/js/wp-sudo-admin-bar.js — keydown handler sets a.style.setProperty('background', '#4caf50', 'important') synchronously, then setTimeout(fn, 300) removes it. a is the .ab-item inside #wp-admin-bar-wp-sudo-active. Guarded by prefers-reduced-motion check (verified). Script is enqueued only when session IS active (mutually exclusive with wp-sudo-shortcut.js).

- Call activateSudoSession(page) to establish an active session.
- Navigate to '/wp-admin/' (admin-bar.js is loaded here — not wp-sudo-shortcut.js).
- Assert #wp-admin-bar-wp-sudo-active is visible (confirms session is active and the right script is loaded).
- Call page.emulateMedia({ reducedMotion: 'no-preference' }) BEFORE pressing the shortcut. This overrides the system/browser prefers-reduced-motion media query so the flash animation runs. Must be called before the keydown event fires (the JS reads matchMedia at handler invocation time).
- Press keyboard.press('Control+Shift+S') — no Promise.all needed, no navigation occurs.
- Immediately evaluate: `document.querySelector('#wp-admin-bar-wp-sudo-active .ab-item')?.style.getPropertyValue('background')` — the inline style is set synchronously in the keydown handler before any setTimeout fires, so the assertion runs within the same event loop turn as the key press.
- Expect the result to be '#4caf50'.

Pitfall to document: The 300ms setTimeout removes the inline styles. The page.evaluate() that reads the style runs in the browser's JavaScript engine after keyboard.press() resolves — this happens well before 300ms, so the style is still present. If timing proves flaky in practice, freeze the setTimeout by installing page.clock AFTER activateSudoSession but BEFORE page.goto('/wp-admin/'). Using page.clock.install() prevents the setTimeout from firing at all, making the assertion unconditionally reliable. The plan includes the simple (no-clock) version first; switch to clock version if CI shows flakiness.
