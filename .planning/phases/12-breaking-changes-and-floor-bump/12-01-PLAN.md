<objective>
Remove the two deprecated governance APIs and add the migration signal:
- BRK-01: hard-remove the `sudo_can()` alias (function must not exist; calling it fatals).
- BRK-02: excise the `compatibility` governance mode from both `wp_sudo_can()` and `wp_sudo_map_governance_meta_cap()`; a stale `compatibility` option falls through to strict.
- BRK-03: add a persistent (non-dismissible) admin notice + `_doing_it_wrong()` that fires on admin screens when the stale option is still `compatibility`, gated on `wp_sudo_can('manage_wp_sudo')`.

Purpose: These are the defining breaking changes of v4.0.0 — they simplify the capability-check contract to a single strict path with `WP_SUDO_RECOVERY_MODE` as the sole break-glass, while signalling (not deleting — that is Phase 13) any stale option left over from a 3.x install.

Output: A simplified `functions-governance.php`, a new `Admin::render_compatibility_mode_notice()` method wired to admin/network admin notices, and updated/added unit + integration tests with the 7 compatibility-mode tests removed.

ASSUMPTION (state in SUMMARY): This phase does NOT bump `WP_SUDO_VERSION` (currently 3.4.0). The `_doing_it_wrong()` version argument is the literal string `'4.0.0'` per RESEARCH — that is the version where the option BECAME wrong, not the plugin's current release version. The release-version bump to 4.0.0 belongs to the release/tagging step, not this phase.
</objective>
