# Phase 10 — Settings UI Revision

## Goal

Restructure the WP Sudo settings page for clarity and usability: add tabbed
navigation, simplify the preset interface, reorder sections, fill help panel
gaps, and improve admin notice language.

## Current State

The settings page (`class-admin.php`) is a single monolithic page rendered by
`render_settings_page()` (line 891). All content — configuration form, MU-Plugin
installer, gated actions reference table, and Rule Tester — renders sequentially.
The preset interface uses radio buttons with per-preset policy lists and a
confirmation checkbox. The help panel has 10 tabs but omits Policy Presets, the
Rule Tester, and Connectors API coverage.

## Pre-phase fixes already applied

### Rule Tester scroll-to-result (committed separately)

The Rule Tester form POST reloaded the page at the top, requiring the user to
scroll down to see results. Fixed by:

1. Adding `id="wp-sudo-tester-result"` to the results `<div>`
2. Appending `#wp-sudo-tester-result` to the form action URL

On POST, the browser scrolls to the results block. When no result exists (no
submission), the anchor target is absent and the page loads normally at the top.

### Task 1: Tabbed navigation infrastructure

Add tab routing to `render_settings_page()`.

**Tab definitions:**

| Key         | Label          | Default |
|-------------|----------------|---------|
| `settings`  | Settings       | Yes     |
| `actions`   | Gated Actions  | No      |
| `tester`    | Rule Tester    | No      |

**Implementation:**

1. Read `$active_tab` from `$_GET['tab']` (sanitized, defaulting to `settings`).
2. Render `<h2 class="nav-tab-wrapper">` with one `<a>` per tab. Each link
   uses `add_query_arg( 'tab', $key, $page_url )`. Active tab gets
   `nav-tab-active` class.
3. Below the tab bar, dispatch to the active tab's render method:
   - `settings` → existing form (reordered per Task 2)
   - `actions` → `render_gated_actions_table()`
   - `tester` → `render_request_rule_tester()`
4. The page title `<h1>`, description paragraph, and preset notice render
   above the tab bar (they apply to the page, not a specific tab).
5. Multisite: tab links use `network_admin_url()` base.

### Task 2: Reorder Settings tab sections

Move the Early Gate (MU-Plugin) section inside the Settings tab, above the form.

**New Settings tab order:**

1. Early Gate (MU-Plugin) — `render_mu_plugin_status()`
2. `<form>` start
3. Policy Presets section (simplified per Task 3)
4. Session Settings section
5. Entry Point Policies section
6. Submit button
7. `<form>` end

**Rationale:** The MU-Plugin is the first decision a new user must make (install
it or not). Placing it first avoids the user having to scroll past configuration
to find it. It remains outside the form since it has its own AJAX handlers.

**Files:** `includes/class-admin.php`

### Task 3: Simplify Policy Presets to dropdown

Replace the radio-button + checkbox interface with a single `<select>` dropdown.

**Changes to `render_field_policy_presets()`:**

1. Remove radio buttons and per-preset policy lists.
2. Render a `<select>` with options: Normal, Incident Lockdown, Headless Friendly,
   and a disabled "Custom" option (shown when current config doesn't match any
   preset).
3. Below the dropdown, show the selected preset's description in a
   `<p class="description">` paragraph. On initial load, this is the stored
   preset's description. Dynamic update on change is a nice-to-have (JS) but
   not required — the server-rendered description for the saved preset is
   sufficient.
4. Remove the "Current configuration:" indicator (the dropdown's selected
   state replaces it).
5. Remove the confirmation checkbox entirely.

**Changes to `sanitize_settings()`:**

1. Remove the `apply_policy_preset` checkbox check.
2. Instead, compare submitted `policy_preset_selection` against the stored
   value. If different and valid, apply the new preset's policies.
3. If the preset selection is unchanged but individual policy fields differ
   from the stored preset's values, mark the preset as `custom`.
4. Remove the `SETTING_APPLY_POLICY_PRESET` constant references.

**Optional JS enhancement (can be deferred):**

Add a small handler in `wp-sudo-admin.js` that swaps the description paragraph
when the dropdown selection changes, using a `data-description` attribute on
each `<option>`.

**Files:** `includes/class-admin.php`, optionally `admin/js/wp-sudo-admin.js`

### Task 6: Simplify preset-applied admin notice

Replace the technical notice format with plain language.

**Current format (line 1710):**
```
Applied the Headless Friendly preset. New surface policies:
rest_app_password_policy: Unrestricted, cli_policy: Limited,
cron_policy: Limited, xmlrpc_policy: Disabled, wpgraphql_policy: Unrestricted
```

**New format:**
```
Headless Friendly preset applied. REST and GraphQL are now unrestricted;
CLI and Cron are limited; XML-RPC is disabled.
```

**Implementation in `render_policy_preset_notice()`:**

1. Replace the key-value iteration with a grouped summary builder.
2. Group policies by value (Unrestricted, Limited, Disabled).
3. Map setting keys to short surface names:
   - `rest_app_password_policy` → "REST"
   - `cli_policy` → "CLI"
   - `cron_policy` → "Cron"
   - `xmlrpc_policy` → "XML-RPC"
   - `wpgraphql_policy` → "GraphQL"
4. Build a sentence: "{surfaces} {are/is} now {value}" for each group,
   joined by semicolons.
5. Format: `"{Preset label} preset applied. {grouped summary}."`

**Edge case:** If all surfaces share the same value (e.g., Normal = all
Limited), simplify to: "Normal preset applied. All surfaces are now limited."

**Files:** `includes/class-admin.php`

**Tests:**
- `test_preset_notice_uses_plain_language_surface_names`
- `test_preset_notice_groups_by_policy_value`
- `test_preset_notice_simplifies_when_all_same_value`

## Risk Notes

- **Settings API compatibility:** `do_settings_sections()` renders all
  registered sections. On the Gated Actions and Rule Tester tabs, the form
  and `do_settings_sections()` must NOT be called. The tab dispatch already
  handles this since only the `settings` tab renders the form.
- **Multisite form action:** The multisite form posts to
  `network_admin_url('edit.php?action=wp_sudo_settings')`. Tab state must
  survive the POST-redirect cycle. The redirect URL (line 867) should include
  `&tab=settings` if not already default.
- **Bookmark URLs:** Tab links should be stable URLs so users can bookmark
  e.g., `?page=wp-sudo&tab=tester` directly.
- **Screen options / help panel:** Help panel is registered once via
  `add_help_tabs()` on the `load-{page}` hook. All 12 tabs show regardless
  of which settings tab is active. This is standard WordPress behavior — help
  covers the whole page.
- **Preset auto-detection on save:** With the checkbox removed, the
  sanitize logic must carefully handle the case where the user changes a
  preset AND also manually edits a policy field in the same save. The preset
  should win (it overwrites), and the notice should reflect the preset, not
  the manual edits.
