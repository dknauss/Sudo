## Phase Boundary

Two independent cleanup tracks folded into one reviewable phase, inserted between Phase 13 (migration/governance audit) and Phase 14 (WordPress.org readiness):

1. **Access-tab UX polish** — make the Grant Capability form on Settings → Sudo → Access operator-friendly. This is the follow-up explicitly deferred in PR #88 ("ROADMAP Access-tab polish: user picker + plain-English capability labels"), now merged to `main` (squash `6abd333`, 2026-06-19).
2. **CI test-speed** — reduce CI wall-clock and runner-minutes for `dknauss/Sudo`, the work scoped in spawned task `task_671a7d54` ("Speed up CI test matrix (fan-out + setup)").

The two tracks touch disjoint files (admin PHP/JS vs. `.github/workflows/` + `bin/`), so they should be **separate plans/PRs within this one phase** — the CI-speed task itself specifies "as its own reviewable PR, separate from feature work."

**In scope:** presentation-layer changes to the Access-tab grant form; CI workflow trimming/caching/rebalancing; updating the #88 E2E regression spec to the new markup.

**Explicitly NOT in scope:** any change to server-side authorization (nonce, `wp_sudo_can`, `GOVERNANCE_CAPS` validation, the grant/revoke/session AJAX handlers' logic); new gating surfaces; new test *assertions of intent* (only selector/markup updates and matrix reshaping).

<decisions>
## Implementation Decisions (locked)

### Access-tab: user picker (ACC-01)
- Replace the numeric `<input type="number" id="wp-sudo-grant-user">` (`includes/class-admin.php` ~line 1409, inside `render_*` for the Access tab) with a **searchable user picker populated from real WordPress users**.
- **Default scope: administrator-role users only** — only admins hold Sudo governance caps in practice. Leave wider eligibility (e.g. a custom `manage_options` role receiving `view_wp_sudo_activity`) as a **documented open question**, not implemented now.
- Client-side validation must reject empty/invalid selection before the AJAX call.
- **Server handler is unchanged**: `$target_user_id = (int) $_POST['user_id']` (~line 1529/1579) keeps working — the picker must submit the user ID in the same field/shape the existing JS + handler expect.

### Access-tab: plain-English capability labels (ACC-02)
- The capability `<select id="wp-sudo-grant-cap">` (~lines 1418–1422) currently renders raw slugs via `foreach self::GOVERNANCE_CAPS`. Replace the **display text** with plain English; **keep `value` = exact slug**.
- Demote the raw slug to secondary text / tooltip so integrators can still identify the capability unambiguously.
- Label mapping (from `docs/ROADMAP.md` §"access tab UX"):
  | Capability slug | Display label |
  |---|---|
  | `manage_wp_sudo` | Manage Sudo settings and policies |
  | `view_wp_sudo_activity` | View sudo activity and sessions |
  | `export_wp_sudo_activity` | Export sudo activity data |
  | `revoke_wp_sudo_sessions` | Revoke other users' active sessions |
- `GOVERNANCE_CAPS` is defined once at `includes/class-admin.php:89`. The label map should live alongside it (single source) — the same labels can be reused by the drift panel and revoke list if helpful.

### Access-tab: contract & test integrity (ACC-03)
- Presentation-only: the AJAX action names, nonce names (per-element `wp_sudo_access` `data-nonce`), and the `admin/js/wp-sudo-admin.js` Access module's request shape must remain compatible.
- The #88 E2E grant-flow regression spec (committed in `145ab8f`, now on `main`) asserts the grant click-through. It will need **selector/markup updates** for the new picker + labelled options, and must stay green. Do not weaken what it asserts (active session → grant lands; no session → `sudo_required`, no grant).
- Preserve a11y: `<label for>` associations and `wp.a11y.speak` announcements (the enqueue already declares the `wp-a11y` dep).

### CI test-speed (CIS-01/02/03) — measured on PR #86 head `b277c4c`, 2026-06-17
The tests are NOT slow (unit ~4s/810 tests; integration runs in seconds). CI cost is **fan-out + per-job setup**. Three levers, each an explicit coverage-vs-speed decision that must be documented:
- **CIS-01 — trim integration matrix.** 10 lanes today (verified against `phpunit.yml`: 5 MS=false + 4 MS=true + 1 MariaDB LTS), ~1m each, dominated by setup. Reduce to a representative subset: retain the MS=false lanes, reduce MS=true to **one** representative combo (first verify which integration tests actually exercise multisite via `grep is_multisite/WP_MULTISITE tests/Integration`), keep the MariaDB LTS lane for DB-engine coverage (~7 lanes). Biggest runner-minute saver. **Document what coverage each dropped lane represented.**
- **CIS-02 — cache the WP test library.** Composer is already cached; the `bin/install-wp-tests.sh` download/install of WordPress + the test suite is NOT. Add `actions/cache` keyed on the install paths so each lane skips it on a hit.
- **CIS-03 — rebalance E2E shards.** 4 Playwright shards ~4.3–5.3m each; shard 2/4 (`challenge-2fa-ui`) is the 5m19s outlier vs ~4.3m. Rebalance (or add a shard) to lower the critical path; note wp-env spin-up is a fixed floor (diminishing returns). Unit (×3 PHP, 18–23s) and static analysis (23–38s) are already fast — **leave alone.**
- Workflow files live in `.github/workflows/` (integration/unit/static matrix is the phpunit workflow; E2E and Plugin Check are separate workflows).

### Claude's Discretion
- Exact user-picker implementation (server-rendered `<select>` of admin users vs. an AJAX typeahead). Decide based on how the existing Access JS reads the user ID and on admin-count scaling; prefer the simplest approach that satisfies "searchable" without new heavy dependencies. WordPress core `wp_dropdown_users()` / the `wp-admin` user-suggest endpoint are candidates to evaluate.
- Whether the capability label map is a new `const`/method on `Admin` or a small helper; whether the slug appears as `<small>` subtitle vs. `title` tooltip vs. both.
- Exact `actions/cache` key strategy and which paths to cache.
- Whether E2E rebalancing is a re-split of specs across shards or a shard-count change.
- Plan/wave structure — but keep Access-tab and CI-speed as separate plans (separate PRs).
</decisions>
