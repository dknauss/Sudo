## Summary

Phase 13.1 folds two disjoint cleanup tracks. Track A (ACC) polishes the Access-tab grant form in `includes/class-admin.php` and its companion `admin/js/wp-sudo-admin.js`. Track B (CIS) trims CI fan-out in `.github/workflows/phpunit.yml` and rebalances groups in `.github/workflows/e2e.yml`. No other files are in scope.

**Track A is presentation-only.** The grant AJAX handler (`handle_grant_cap()`, line 1520) reads `(int) $_POST['user_id']` — the JS post shape is already integer, so a `<select>` of admin users with numeric `value` attributes is a drop-in that requires zero PHP change. The capability `<select>` already has the correct contract (`value` = slug); only the display text and optional slug secondary text change. No new PHP public methods, no new authorization logic, no changes to nonces or AJAX action names.

**Track B is CI config work only.** The integration matrix currently has exactly 10 lanes (not 11 as stated in the CONTEXT.md estimate — the live YAML was verified). The WordPress test library download (SVN export of `tests/phpunit/includes/` and `/data/` plus `wp-tests-config-sample.php`, plus a WP core tarball) is the uncached per-lane setup cost. `actions/cache@v5` is already in use for Composer; the same pattern applies to the test lib. The E2E `access-grant.spec.ts` is orphaned (present in `tests/e2e/specs/` but absent from all four `e2e.yml` group test-command lists) and must be assigned in CIS-03.

**Primary recommendation:** Use a server-rendered `<select>` populated via `get_users(['role' => 'administrator'])` for the user picker — it requires only PHP changes and a minor JS guard update, adds no JS deps, and handles the realistic case of small administrator pools with no extra network round-trips.

### Pattern 1: Server-Rendered User Picker (Recommended)

**What:** Replace `<input type="number">` with `<select id="wp-sudo-grant-user">` populated from `get_users(['role' => 'administrator', 'orderby' => 'display_name', 'fields' => 'all'])`. Each `<option value="{ID}">{display_name} ({user_login})</option>`.

**Why preferred over AJAX typeahead:**
- The Access tab already calls `get_users()` twice (drift panel at line 1451, cap holders at line 1682) — a third `get_users` call for admin users is consistent and adds no new patterns.
- Administrator pools are small (typically 1–10 users). A static `<select>` is perfectly usable without typeahead for this cardinality.
- No new JS dependencies. The existing Access module only needs a one-line change: `parseInt(userInput.value, 10)` already works on `<select>` elements — `select.value` is a string that `parseInt` converts correctly.
- The `<select>` element is natively "searchable" in the browser via keyboard matching (type a letter to jump to the matching option). This satisfies "searchable" without custom JS.
- `wp_dropdown_users()` is an alternative (outputs a `<select>` with `role__in => ['administrator']` support), but generates a full `<option>` block — using `get_users()` directly gives identical output with explicit control over the `<option>` text format (login + display_name).

**The only JS change needed:** Replace `userInput.focus()` in the validation block (JS line 369) — the input element reference still works since we're changing the PHP element type from `<input>` to `<select>`, both selected by `document.getElementById('wp-sudo-grant-user')`. The `parseInt(userInput.value, 10)` parsing at JS line 360 works identically for `<select>` because `select.value` returns the value of the selected `<option>`. Guard logic `!userId || userId < 1` is unchanged.

**Multisite note:** `render_access_tab()` is already multisite-aware (it calls `get_users()` which is blog-scoped). The user picker should scope to the current blog's administrators on multisite — standard `get_users()` behavior, no extra work.

**Example:**
```php
// Source: pattern derived from existing get_users() calls in class-admin.php
$admin_users = get_users(
    array(
        'role'    => 'administrator',
        'orderby' => 'display_name',
        'order'   => 'ASC',
        'fields'  => 'all',
        'number'  => -1,
    )
);
```
```html
<select id="wp-sudo-grant-user">
  <option value="0"><?php esc_html_e( '— Select a user —', 'wp-sudo' ); ?></option>
  <?php foreach ( $admin_users as $u ) : ?>
    <option value="<?php echo (int) $u->ID; ?>">
      <?php echo esc_html( $u->display_name . ' (' . $u->user_login . ')' ); ?>
    </option>
  <?php endforeach; ?>
</select>
```

**Label update:** The `<label for="wp-sudo-grant-user">` currently reads "User ID". Change to "User" (the label describes what you're selecting, not the identifier format).

**Localized string update:** `wpSudoAdmin.access.invalidUser` is currently "Enter a valid user ID." — update to "Select a user." The JS validation guard (`!userId || userId < 1`) catches the default `value="0"` placeholder option, so the existing guard works unchanged.

### Pattern 2: Plain-English Capability Labels (ACC-02)

**What:** Introduce a label map constant or method alongside `GOVERNANCE_CAPS`. Use it in `render_access_tab()` to render `<option value="{slug}">{label} <small>{slug}</small></option>`. Also reusable in the drift panel's "Grant manage_wp_sudo" button if desired.

**Where the map lives:** A private constant `CAP_LABELS` (array) on `Admin` declared next to `GOVERNANCE_CAPS` at line 89 is the simplest approach. Alternatively a private static method `cap_label(string $cap): string` doing a `match`. Either satisfies "single source."

**Slug as secondary text:** HTML `<option>` elements do not render child HTML — `<small>` inside `<option>` is ignored by browsers. Use the `title` attribute for tooltip (secondary, not always visible) and also include the slug parenthetically in the option text for in-list visibility:

```html
<option value="manage_wp_sudo"
        title="manage_wp_sudo">
  Manage Sudo settings and policies
</option>
```

Or combine both display text and slug:
```html
<option value="manage_wp_sudo">
  Manage Sudo settings and policies (manage_wp_sudo)
</option>
```

The `(slug)` in parentheses approach is the most universally visible (works in all browsers including accessibility tools that ignore `title`). CONTEXT.md says "demote the raw slug to secondary text" — parenthetical or `title` are both valid; parenthetical is more reliable.

**`<select>` changes only affect display; `value` stays the exact slug** — no change to the AJAX contract or `GOVERNANCE_CAPS` validation.

### Pattern 4: WP Test Library Caching (CIS-02)

**What `bin/install-wp-tests.sh` does (verified):**
- Downloads WP core tarball to `/tmp/wordpress` (`$WP_CORE_DIR`)
- SVN-exports `tests/phpunit/includes/` and `tests/phpunit/data/` to `/tmp/wordpress-tests-lib` (`$WP_TESTS_DIR`)
- Downloads `wp-tests-config-sample.php` and patches it with DB credentials
- Downloads Two Factor plugin to `/tmp/wordpress/wp-content/plugins/two-factor/`

**What can be cached:** `/tmp/wordpress` and `/tmp/wordpress-tests-lib`. These are pure downloads — DB credentials are injected at install time into `wp-tests-config.php`, but the SVN `includes/` and `data/` dirs and the core tarball are deterministic per WP version.

**What cannot be cached:** The `wp-tests-config.php` file contains DB credentials patched by `sed`. This file lives inside `/tmp/wordpress-tests-lib/` and will be wrong on a cache hit. The cache-restore step must run `install_wp_tests.sh` again (it will skip the SVN export if the directory exists via `[ -d $WP_TESTS_DIR ]` guard at line 246) but will re-patch `wp-tests-config.php` and re-create the DB. Alternatively: skip caching `wp-tests-config.php` and only cache the `includes/` and `data/` subdirs.

**Simplest approach:** Cache `/tmp/wordpress-tests-lib/includes` and `/tmp/wordpress-tests-lib/data` separately from `wp-tests-config.php`. Or cache the whole `/tmp/wordpress-tests-lib` dir but always regenerate `wp-tests-config.php` afterwards (the script's `if [ ! -s "$WP_TESTS_DIR/wp-tests-config.php" ]` guard at line 255 means it only writes config if absent — **remove the config from the cache** using `save-always: false` and explicit paths, or delete it before saving).

**Recommended cache key pattern:**
```yaml
key: wp-tests-lib-${{ matrix.wp }}-${{ hashFiles('bin/install-wp-tests.sh') }}
restore-keys: wp-tests-lib-${{ matrix.wp }}-
```
The WP version is the primary cache discriminator. `hashFiles('bin/install-wp-tests.sh')` invalidates if the install script changes.

**Paths to cache:**
```
/tmp/wordpress-tests-lib/includes
/tmp/wordpress-tests-lib/data
/tmp/wordpress
```
After restore, still run `install-wp-tests.sh` to handle DB creation and config patching — the script's directory-existence guards skip re-downloading if dirs exist.

**Install SVN step:** The `svn export` is only called if the directory does not exist. On cache hit, the dirs exist and SVN is not invoked. The `sudo apt-get install -y subversion unzip` step can remain but costs only ~5s (apt cache is hot in ubuntu-24.04).

### Anti-Patterns to Avoid

- **Adding Select2/SelectWoo/Chosen for the user picker:** No JS library of this kind is bundled in WordPress core (core bundles Select2 only in the full admin asset bundle, not as a standalone dep available to plugins). Adding an external selectize library would be a new dependency. Avoid.
- **Using WordPress's `wp-ajax-user-search` or `user-search.php` endpoint:** This exists in WP core admin but is not a public API, is not documented for plugin use, and requires enqueuing `user-search` handle — more brittle than a server-rendered `<select>`.
- **Adding new public PHP methods just for JS testability:** Per CLAUDE.md "No Test-Environment Shims" — don't add public getters or render helpers solely to make unit testing easier. The existing unit test for `render_settings_page` (line 2992) asserts 'Access Control' in output — if it needs to assert the new `<select>` markup, use `assertStringContainsString('wp-sudo-grant-user', $output)`.
- **Caching `wp-tests-config.php`:** This file contains patched DB credentials. Always regenerate it; only cache the WP test library source files.
- **Weakening the E2E spec assertions:** ACC-03 locks that the `access-grant.spec.ts` assertions (sudo_required on no-session, grant succeeds on session) must not be weakened. Only selector paths need updating (e.g., `page.fill('#wp-sudo-grant-user', ...)` → `page.selectOption('#wp-sudo-grant-user', ...)` and `page.selectOption('#wp-sudo-grant-cap', 'manage_wp_sudo')` can likely stay since cap `value` attrs are unchanged).
- **Changing integration matrix without documenting dropped coverage:** CIS-01 requires explicit documentation of what each dropped lane represented.

## Common Pitfalls

### Pitfall 1: `page.fill()` on `<select>` in E2E

**What goes wrong:** `access-grant.spec.ts` line 94 uses `page.fill('#wp-sudo-grant-user', String(targetId))`. After ACC-01 changes the element from `<input type="number">` to `<select>`, `page.fill()` will throw — `fill()` only works on input/textarea/contenteditable. Must replace with `page.selectOption('#wp-sudo-grant-user', String(targetId))`.

**How to avoid:** Update `access-grant.spec.ts` lines 94, 114 from `page.fill` to `page.selectOption`. The E2E spec must also ensure the `targetId` option actually exists in the `<select>` — the `e2e_grant_target` user created in `beforeAll` is an editor (not admin), so it will NOT appear in an admin-scoped `<select>`. The test setup must either: (a) grant the editor the administrator role, or (b) create the target as an administrator. The test currently works with the server handler accepting any user ID — after ACC-01 the server handler is unchanged but the picker only shows administrators. The E2E spec must adapt.

**Warning signs:** `page.fill() is not supported for <select>` error in Playwright CI.

### Pitfall 2: `wp-tests-config.php` credentials on cache restore

**What goes wrong:** If `/tmp/wordpress-tests-lib` is cached including `wp-tests-config.php`, the cached config has credentials from the original run's environment (DB host/user/pass). On cache restore in a fresh job, these credentials are wrong, causing integration test DB connection failures.

**How to avoid:** Either exclude `wp-tests-config.php` from the cached path (cache only the `includes/` and `data/` subdirs) or always re-run the credential-patching portion of `install-wp-tests.sh`. The script's guard at line 255 (`if [ ! -s "$WP_TESTS_DIR/wp-tests-config.php" ]`) will skip writing config if the file already exists and is non-empty — so if credentials from a prior run are in the cache, config won't be regenerated. Best: delete `wp-tests-config.php` after restore, before running the install script, so the script regenerates it.

### Pitfall 3: `parseInt()` on `<select>` value — already correct, but verify default option

**What goes wrong:** The JS validation `!userId || userId < 1` (line 362) correctly catches `userId = 0` (the placeholder `<option value="0">`). If the `<select>` has no placeholder and the first real admin user is pre-selected, the validation passes immediately without user interaction — confusing UX. The placeholder `<option value="0">` approach is correct and required.

**How to avoid:** Always include a `<option value="0">— Select a user —</option>` as the first option so the initial state fails validation and forces an explicit selection.

### Pitfall 4: E2E E2E `access-grant.spec.ts` not in any CI group — spec is silently never run

**What goes wrong:** `access-grant.spec.ts` exists in `tests/e2e/specs/` but is not in any `test-command` in `e2e.yml`. CI passes without running it. This is already the current state on `main` after PR #88.

**How to avoid:** CIS-03 must assign the spec to a group. This is a required fix, not optional rebalancing.

### Pitfall 5: Unit test for `render_settings_page` may need updated assertion

**What goes wrong:** `tests/Unit/AdminTest.php:2992` (test `test_render_settings_page_renders_access_tab`) stubs `get_users()` to return `[]` and asserts `'Access Control'` in output. After ACC-01, `render_access_tab()` calls `get_users(['role' => 'administrator'])` for the picker. The stub returns `[]` → the `<select>` has only the placeholder option — no runtime error, but the test should be extended to assert the picker `<select>` exists. If `get_users()` is not stubbed for the new call, Brain\Monkey will throw (undefined WP function). The test MUST stub the new `get_users` call.

**How to avoid:** When updating `render_access_tab()` to call `get_users(['role' => 'administrator'])`, update the unit test to stub this new call and assert the resulting `<select>` element is present.

### Pitfall 6: `docs/current-metrics.md` must be updated BEFORE commit

**What goes wrong:** CLAUDE.md requires updating `docs/current-metrics.md` FIRST if any counted surface changes. CIS-01 changes the integration lane count (from 10 to the trimmed number). The CI Matrix Snapshot section at line 63 currently reads "PHP 8.0, 8.1, 8.3" (stale — doesn't match the verified PHP 8.2/8.3 in the live matrix). CIS-03 changes the E2E group count if a 5th group is added. Both require `current-metrics.md` updates before committing the workflow changes.

   - What we know: The test creates `e2e_grant_target` as `--role=editor`. After ACC-01, the picker only shows administrators.
