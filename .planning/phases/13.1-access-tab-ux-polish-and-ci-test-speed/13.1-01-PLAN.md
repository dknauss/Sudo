<objective>
Make the Settings → Sudo → Access **Grant Capability** form operator-friendly: replace the raw numeric user-ID `<input type="number">` with a server-rendered `<select>` of administrator-role users, and replace the capability dropdown's raw-slug display text with plain-English labels (slug demoted to secondary text). This is the PR #88 follow-up.

**Purpose:** Operators no longer need to know user IDs or memorize capability slugs to grant governance capabilities.

**Output:** Presentation-only changes to `includes/class-admin.php` + `admin/js/wp-sudo-admin.js`, an updated render unit test, and an updated #88 E2E spec — all green. **No server-side authorization, nonce, AJAX action name, or `GOVERNANCE_CAPS` validation changes.**

CRITICAL CONSTRAINT (hard requirement, encoded as a must_have): this change is presentation-only. The grant handler `handle_grant_cap()` (~line 1520) reading `(int) $_POST['user_id']`, the `check_ajax_referer('wp_sudo_access', '_nonce')` nonce, the AJAX action names, and the JS request shape MUST remain byte-compatible. The `<option value>` attributes in the capability select MUST stay the exact capability slugs.
</objective>
