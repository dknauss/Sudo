<objective>
Wire the UI revocation surfaces on the Users list, consuming the shared method,
shared liveness predicate, and factored core from plan 01: a per-user "Revoke sudo
session" row action (no-JS classic nonced link → admin_post_ handler → redirect with
dismissible notice), the revoke-all button + interstitial confirm screen, and the
three distinct blocked-path messages plus success — all via wp_admin_notice(). No new
JS is added to users.php (it has none today).

Purpose: Put revocation where admins already manage users and where active sessions are
already visible ("Sudo Active (N)"), keyed on the browser-independent
`_wp_sudo_expires > time()` enumeration via the shared Sudo_Session::is_session_live()
predicate from plan 01 — NOT Sudo_Session::is_active() (cookie-bound). Separate session
revocation from capability administration (the core intent of REVK-04, whose removals
land in plan 03).

Output: Row-action visibility gate, admin_post_ handler(s) (per-user + revoke-all
confirm/perform), the Users-list revoke-all button + interstitial, and correct message
selection — including the target-already-expired race path — all under TDD.

REVK-02 scope note (deviation, authoritative per 24-CONTEXT.md): revoke-all is
CURRENT-SITE scoped for v4.5 (the button renders on each site's Users list and acts on
that site's sessions). REQUIREMENTS.md REVK-02 says "network-wide"; true cross-site
enumeration is deferred as REVK-F3. This is deliberate — see plan 01 for the full note.
</objective>

From plan 01 (includes/class-admin.php) — factored core to call:
```php
// Returns an outcome struct. Outcomes: no_cap | self_target | target_expired | rate_limited | success.
// Check order (objection A, cap-first so target state never leaks to a capless caller):
//   cap → self_target (target===revoker) → target-liveness → rate limit → deactivate → audit hook.
// An already-expired target yields 'target_expired' with NO deactivate() and NO rate slot
// consumed. The single $reason param IS the surface tag (no separate $surface) and flows to
// the wp_sudo_session_revoked audit hook. The caller maps the outcome to a redirect query
// arg + wp_admin_notice().
private function revoke_session_core( int $target_user_id, int $revoker_user_id, string $reason ): array;
```

Existing Users-list integration already present (includes/class-admin.php):

add_filter( 'views_users', array( $this, 'filter_user_views' ) );

Operator-self exclusion: get_current_user_id() vs the row's user; never render on own

wp_admin_notice() (WP 6.4+, already used in plugin) — render via the redirect query
